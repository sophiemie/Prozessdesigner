Preface

Doxygen is a mighty tool for documentation source codes written in various programming languages - but unfortunately not for Delphi/Pascal. Nevertheless it is no problem to document Delphi sources with Doxygen comments preceded with "//!" or "///" since Delphi treats "//" as line comment as C/C++ does.

But since Doxygen can't parse Delphi sources there is the question how to create the Doxygen documentation for the Delphi entities.

The obvious solution is to exclude Delphi source codes from parsing by Doxygen. This is easily done by surrounding them with \cond .. \endcond like this simple Delphi unit example:

//! @file
//! @brief this is a Delphi source code unit
//! ...
//! @cond
unit example;
interface
implementation
end;
//! @endcond

The first lines up to @cond would appear in Doxygen's files section documentation while the whole following Delphi code is ignored.

Next idea is to put further comments to the source code while first enabling and afterwards disabling Doxygen parsing by \endcond and \cond:

//! @endcond
//! @brief this is my first procedure
//! @cond
procedure myproc(),
begin
end;

But because Doxygen can't parse Delphi sources it doesn't find the corresponding function "myproc()" and the Doxygen comment is ignored (or placed also in the files section)!

At this point Delphi's C++Builder ability to create C++ header files takes place. How to use this feature for Doxygen is described in the following chapters

Additionally it would be nice to insert the particularly Delphi and/or C++ declaration into the Documentation. This is provided by a couple of aliases, also described in the following chapters.

Introduction

Because Delphi only provides the C++ (.hpp)-headers but not the C++ (.cpp) source modules not all possibilities of Doxygen are available.

Abilities:
  • Documenting most entities like C/C++
    • types
    • variables
    • functions / procedures
    • classes inc. methods, fields and properties
  • Unit dependencies are shown as include file dependencies
  • Class dependencies and collaboration
Restrictions:
  • The following Doxygen features are not available because there are no .cpp source modules Doxygen could parse:
    • Function dependencies
    • call / caller graphs
  • overloaded functions / procedures are not allowed (may not documented)
  • functions / procedures in local function / procedure scopes are not recognized
Conventions
  1. All of the special aliases must be notated before the respective declaration
  2. Since Delphi is (in the opposite of many other languages i.e. C/C++) not case sensitive the following conventions should be met:
    • reserved words used in the ALIASES must be written in all lower case letters, for details refer to Documenting Delphi source code
    • care must be taken to always use the same spelling of identifiers
    • formatting conventions for some entities, s. Special ALIASES

TExample1

As described above one can document the entities directly with Doxygen's special commands; additionally it's convenient to include the respective Delphi declaration, which can be done using \snippet:

Delphi code:
20  //! @class TExample1
21  //! @brief Sample class1
22  //!
23  //! @details
24  //! @par Delphi Declaration - included with \\snippet command:
25  //! @snippet Example_UNIT.pas TExample1
26  //! @cond
27  // [TExample1]
28  TExample1 = class(TObject)
29  Var1: INTEGER;
30  FUNCTION Func1(): INTEGER;
31  PROCEDURE Proc1();
32  END;
33  // [TExample1]
Resulting documentation:
TExample1

TExample2

Since this requires two additional block IDs, the snippet between TExample1 = and the closing END; my be inserted directly via special command \skip and \until:

\skip TExample1 =
\until END;

But this would also require to put the documentation behind the entity because the commands are in the same file we're looking in: Doxygen's include mechanism would find the both markers "TExample1 =" and "END;" at lines "\\skip" and "\\until".

So there are two things to be done (TExample2):

  • Packing this command into a more general alias to avoid the explicit notation of this two markers.
  • Creating a new Snippet command which gets two markers

And since "END;" is

  1. not very specific to mark the end of a class-declaration (may also appear inside the declaration)
  2. case sensitive (Delphi allows all other spellings!)

the end of the declaration is marked with a special comment " // TExample2" after "END;". Thus only the name of the entity (TExample2) is required as argument:

Delphi code (TExample2):
37  //! @endcond
38  //! @class TExample2
39  //! @brief Sample class2
40  //!
41  //! @details
42  //! @par Delphi Declaration - included with special alias \\DCompTypeDecl:
43  //! @dontinclude Example_UNIT.pas
44  //! @DCompTypeDecl{TExample2}
45  //! @cond
46  TExample2 = class(TObject)
47  Var2: INTEGER;
48  FUNCTION Func2(): INTEGER;
49  PROCEDURE Proc2();
50  END; // TExample2
Resulting documentation:
TExample2
With two aliases
ALIASES += \
"DCompTypeDecl{1}=\Snippet{\1 =,; // \1}" \
and
ALIASES += \
"Snippet{2}=\skip \1 ^^ \until \2 ^^ " \

TExample3

The third example summarizes these recurring steps in one special command (\DClass):

  1. Starting a documentation block (i.e. \class)
  2. Inserting an anchor to be able to link to the Delphi code documentation
  3. Inserting a headline (\par Delphi documentation)
  4. Preloading the Delphi module file (\dontinclude *.pas)
  5. Skipping to the beginning of the entity's declaration
  6. Inserting the Delphi code

A note on step 4. (Skipping): Finally the command doesn't skip to the beginning of the Delphi declaration but to the first argument "{TExample3," of the special alias (i.e. "\\DClass"). This is more specific particularly in this case: This avoids errorneous skipping in case there is a forward class declaration TExample = class;!

Delphi code (TExample3):
54  //! @endcond
55  //! @DClass{TExample3,Example_UNIT}
56  //! @brief Sample class3
57  //! <b>Delphi code included with special command \\DClass</b>
58  //! @cond
59  TExample3 = class(TExample2) //! This Doxygen comment doesn't appear anywehre!
60  Var3: INTEGER; // an integer variable
61  FUNCTION Func3(): INTEGER; // a function declaration
62  PROCEDURE Proc3(); // a procedure declaration
63  END; // TExample3
Resulting documentation:
TExample3

In this example the \DClass{TExample3,Example_UNIT} command alias expands to:

  • \class TExample3 "" ""
    activates a class section
  • \anchor TExample3_pas
    inserts an anchor to be able to link to this Delphi code documentation
  • \par Delphi-Declaration
    starts a detailed documentation section with heading "Delphi-Declaration"
  • \dontinclude Example_UNIT.pas
    sets Example_UNIT.pas (which must contain the class declaration) as include file
  • \skip {TExample3
    skips to the special command with first argument (note the leading { !) "TExample3". This prevents wrong skipping in case there is a forward declaration "<tt>class = TExample3;</tt>" before the \DClass command.
  • \skipline TExample3 =
    searches the line with the class declaration and inserts this line...
  • \until ; // TExample3
    ...up to the line containing the end of declaration

All together this activates a class section and inserts the Delphi class declaration as detailed description.

After that the class can documented by Doxygen means using Doxygen special commands (including \brief which is correctly moved by Doxygen to the top of documentation!). The documented entity may be followed by any additional Delphi comment which becomes not part of the Doxygen documentation.

A note on Comments

  • Doxygen comments outside a \endcond ... \cond section dont't appear anywhere in the documentation.
  • Particularly if Doxygen includes code snippets those comments are also filtered, but not "normal" comments, s. the above TExample3: Compare the Delphi code against the resulting documentation.
